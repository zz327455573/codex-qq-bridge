## 数据流

### 正向流（QQ → Codex）

1. QQ 用户发送私聊消息
2. QQ 官方网关通过 WebSocket 推送 `C2C_MESSAGE_CREATE` 事件
3. 桥接器 `event_loop` 接收事件 → 提取 `content` + `attachments`
4. 命令检查：`/new` → 重启 Codex；`/stop` → 停止 Codex
5. 普通消息 → `send_to_codex()` → `tmux send-keys -t codex "消息" Enter`

### 反向流（Codex → QQ）

1. Codex 执行后输出到终端 → 写入 `rollout-*.jsonl` 日志
2. 桥接器 `jsonl_poll` 协程（每 2 秒轮询）检测文件变化
3. 增量读取新行 → `parse_codex_line()` 提取 `assistant` 回复
4. `send_message_rest()` → QQ REST API POST `/v2/users/{openid}/messages`

## 关键设计决策

### 为什么用 JSONL 轮询而不是 capture-pane？

- capture-pane 会捕获终端控制字符（ANSI 转义码），需要额外清理
- tmux 滚动缓冲区不可靠，多屏内容可能丢失
- JSONL 是 Codex 原生输出的结构化日志，纯文本无控制符
- JSONL 支持增量读取，天然适合边输出边推送

### 为什么没有审批按钮？

- Claude Code 桥需要审批按钮（因为 Claude Code 默认交互式审批）
- Codex 自带 `--ask-for-approval on-request` 模式，自行判断是否需用户确认
- 桥接层不引入审批拦截，保持架构简洁

### 自动 MASTER_OPENID 绑定

- 如果 `.env` 中未设置 `MASTER_OPENID`，第一个发消息的用户自动成为管理员
- 绑定后写入 `.env` 文件持久化，重启后生效
- 多用户场景需手动设置 `MASTER_OPENID` 限制只响应特定用户
